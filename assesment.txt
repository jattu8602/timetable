01 Context
Read this before you start. The why behind the product shapes every decision you'll make in the panel.
Samayak is the academic operations platform built by Anugat AI for Indian colleges and universities.
University timetabling is formally an NP-hard problem — assigning faculty, rooms, subjects, and time slots
without conflicts — and most institutions still do it by hand. Samayak closes that loop across four stages: Plan
(AI-assisted, conflict-free scheduling), Operate (real-time view of planned vs actual), Recover (structured
workflows to reschedule lost classes instead of absorbing them), and Improve (tracking delivery data so the
next semester's plan gets better).
The Admin Panel is the foundation everything else stands on. It is where a university administrator sets up
and manages the core data — departments, rooms, courses, and users — that every scheduling, recovery,
and analytics feature depends on. We already have working interfaces for coordinators, HODs, deans, and
faculty. The Admin Panel is the missing piece.
02 The Task
Design and build the Samayak Admin Panel as a working full-stack application — frontend, backend APIs, and
a real database. Section 5 defines the pages and their capabilities. How you structure, lay out, and present
them is yours to decide.
The most important thing we are evaluating is not whether you built all five pages — it is whether you built
them well. UX is the primary lens. Design-system alignment is the second. Everything else follows.
One specific test: the Dashboard must not be a static summary screen. We will feed your system new data
mid-review (Section 8) and watch how the analytics respond. Build the dashboard as a live analytical
instrument, not a counts panel.

05 Admin Pages to Build
Six pages. Build these well rather than building more. Layout, navigation, and interaction patterns are
yours to design.
5.1 Dashboard / Analytics the admin's analytical instrument — not a summary screen
This is where we most closely evaluate the candidate's product thinking. The dashboard must present
actionable analytics derived from the institution's timetable data — not static counts. The four required
metrics are:
Room Utilisation %
utilisation = (slots occupied / total available slots) x 100 [per room, per day,
overall]
Shows which rooms are over-relied on and which are sitting idle. Displayed per room and as a department-wide
figure. Should update when new data is imported.
Probability of Finding an Empty Room — per Time Slot
P(empty | slot S) = free rooms in slot S / total rooms [for each period I through
IX]
Directly supports class recovery. A coordinator looking for a replacement slot needs to know which time periods
have rooms available. The higher this probability, the safer the recovery window.
Under-Running Courses
under-running = courses where scheduled contact slots < (credit hours implied by
credit count)
Flags courses that are behind their required delivery. A 4-credit lecture course implies 4 contact hours per week; if
only 3 slots are scheduled, it is under-running. The dashboard should list these and quantify the gap.
Average Empty Room-Hours per Day
avg empty room-hours = mean over all rooms of (unscheduled slots x slot duration)
across a working day
Quantifies idle infrastructure. High empty room-hours alongside low room utilisation signals scheduling
inefficiency and wasted capacity.
In addition to these four: the dashboard must react to live data changes. Import a new department and the
metrics should recalculate. Add a room and utilisation should shift. This reactivity is what separates a real
analytical instrument from a static display.
5.2 Departments departments & branches
• Add a department/branch manually with a name and unique short code.
• Bulk import from Excel/CSV with per-row validation and a readable pass/fail report.
• Searchable, paginated list. Edit and delete with confirmation; deleting a department that other records
depend on should warn, not silently break.

Bulk import (Excel/CSV) is a shared capability across pages 5.2–5.5. Design one import experience and reuse it. The
PDF ingestion page (5.6) is a distinct, dedicated flow.

08 Timetable PDF Ingestion Page
This is the unseen-scenario test. It is the most technically demanding page in the assignment.
Build a dedicated page in the admin panel where a reviewer can upload a department timetable in PDF — in
the same format as the CSE data provided — and the system will parse, extract, and integrate it
automatically.
The page must do the following:
1. Accept a PDF upload in the BIT Mesra timetable format. The reviewer drops the file, the system
processes it asynchronously (BullMQ), and the page shows live progress feedback.
2. Extract structured data from the PDF: department name, branch and section identifiers, time slots and
days, course codes and names, room identifiers, and faculty names — mapping each to the correct
entity in the database.
3. Integrate into the system — departments, rooms, courses, and faculty that do not exist are created;
those that already exist are matched and linked, not duplicated. The reviewer sees a clear import
summary: what was created, what was matched, and what could not be parsed.
4. Trigger analytics recalculation — the moment ingestion completes, all four dashboard metrics (room
utilisation, P(empty room) per slot, under-running courses, avg empty room-hours) must recompute to
reflect the newly added department.
5. Error handling (Bonus Point ) – If the uploaded PDF does not fully align with the expected format or is
missing information, the system should degrade gracefully rather than fail entirely. This is an openended evaluation criterion.
Possible approach: The system ingests whatever structured data it can extract, flags what could not be
parsed or was incomplete (missing room capacity, absent faculty assignments, etc.), and creates a
"pending" or "incomplete" state for those entities. The admin can then upload a supplementary PDF or
manually fill gaps later without re-importing the entire dataset.
Your creativity here matters. Build an ingestion flow that is robust to real-world PDF variations —
tables with merged cells, inconsistent column orders, missing sections, or partial data — and
demonstrate how the system handles degradation without losing what it could parse. This is how real
production systems behave.
What is being evaluated: The reviewers will upload a timetable PDF for a department the system has
never seen. How well the system handles that unseen data is the core test. Accuracy of extraction,
graceful handling of parse failures, correct entity matching, and live analytics update are all judged.
Build the parser to be format-aware, not brittle. The table layout, column order, and heading styles in the
CSE PDF are the contract — build to that contract and the system will work for any department's data in
the same format.
The upload PDF will not be shared before submission. Build for the format, not for CSE specifically.
